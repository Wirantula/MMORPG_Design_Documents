# Story 13.1: Test Strategy Implementation

## Problem / intent
The project needs layered tests covering every seeded module so the simulation remains stable as it grows. Without comprehensive test coverage, regressions from parallel agent work will be undetectable until production.

## Acceptance criteria
- Unit tests exist for every server module: env config, command DTO, health controller, simulation service, observability service, realtime gateway, and message envelope contracts.
- Unit tests exist for every client library: config, API client, realtime client, and contract helpers.
- Cross-module contract test validates server/client envelope parity.
- All tests pass via `npm run test` in both `server` and `client` workspaces.
- Test files follow workspace conventions (`server/test/*.test.ts`, `client/src/**/*.test.ts`).

## Dependencies
- 1.1 monorepo/workspace foundation (done)
- 1.3 observability baseline (done)
- 3.1 websocket gateway scaffold (done)
- 3.2 browser shell scaffold (done)
- 4.1 world time service scaffold (done)

## Implementation notes
- Server tests use Vitest with node environment, matching `server/vitest.config.ts`.
- Client tests use Vitest with node environment, matching `client/vitest.config.ts`.
- Cross-module tests live in `tests/` with their own Vitest config.
- Gateway tests use mock sockets rather than real Socket.IO connections (unit-level, not integration).
- Env tests use `vi.resetModules()` + dynamic imports to handle the `loadEnv()` caching behavior.
