# Story 13.3: Security and Exploit Review (Foundation Phase)

## Problem / intent
The foundation codebase needs a security baseline review so abuse vectors are documented, prioritized, and tracked before features built on top of them increase the attack surface.

## Acceptance criteria
- Threat model document exists at `docs/SECURITY_REVIEW.md` covering all seeded code.
- Each finding has a severity rating, status, detail, and recommendation.
- An exploit test checklist is documented for use in every release candidate review.
- A hardening roadmap prioritizes remediation across future epics.
- Observability recommendations for security-relevant metrics are documented.

## Dependencies
- 1.1 monorepo/workspace foundation (done)
- 3.1 websocket gateway scaffold (done)
- 13.1 test strategy (done – delivered together)

## Implementation notes
- Review scope: `server/src/`, `client/src/`, `infra/compose/`, env config, and contract definitions.
- 10 findings documented ranging from 🔴 High (no auth on WebSocket) to 🟢 Low (health info disclosure).
- Exploit test checklist has 7 manual verification steps for release gating.
- Hardening roadmap has 8 items ordered by impact priority.

## Test notes
- The command.dto.test.ts and realtime.gateway.test.ts test files from Story 13.1 cover several abuse vectors:
  - Malformed input rejection (Zod validation)
  - Unknown command type rejection
  - Null/undefined/non-object input handling
  - Unknown field stripping (prototype pollution mitigation)
- Additional exploit tests (WebSocket flood, oversized payload) are documented as manual checklist items for pre-release validation.

## Observability notes
- Security review recommends four new metrics: `cybaworld_ws_connections_total`, `cybaworld_ws_commands_total`, `cybaworld_ws_rate_limited_total`, `cybaworld_ws_errors_total`.
- All security-relevant events should use `context: "Security"` in structured log entries.
