# Story 4.2: Action Engine v1

## Problem / intent
Players need to submit time-consuming actions (rest, forage, train, socialize) that consume in-game time and resolve with outcomes, so daily life has structure and the action loop is functional.

## Acceptance criteria
- A seed catalog of action definitions exists (rest, forage, train-strength, socialize) with category, duration, and repeatable flag.
- A character can start one action at a time; duplicate starts are rejected.
- Actions support `active`, `paused`, `completed`, and `cancelled` states.
- Pausing preserves remaining time; resuming recalculates the end time.
- The tick loop resolves actions whose end time has passed and emits `ActionResolved` domain events.
- WebSocket `action.submit` and `action.cancel` commands route through the gateway to the action engine and return `action.started` / `action.cancelled` events to the client.
- Domain events `ActionSubmitted`, `ActionResolved`, `ActionCancelled` are emitted for all state transitions.
- Deterministic tests cover start, cancel, pause, resume, completion, double-start, cancel-empty, and pause-resume timing.

## Dependencies
- Story 4.1 (world time service) — implemented in this changeset.

## Implementation notes
- `ActionDefinition` and seed catalog in `server/src/modules/simulation/actions/action-catalog.ts`.
- `ActionQueue` (in-memory `Map<characterId, ActionSlot>`) in `action-queue.ts`.
- `ActionService` in `action.service.ts` — startAction, cancelAction, pauseAction, resumeAction, tickActions.
- `RealtimeGateway` routes `action.submit` / `action.cancel` to `ActionService` and emits typed server events.
- Contracts updated: `action.cancel` added to `ClientCommandType`; `action.started`, `action.resolved`, `action.cancelled` added to `ServerEventType`.
- Rewards are placeholder (`{}`); the progression engine will fill these in later stories.

## Test notes
- `test/action.service.test.ts` — 16 tests covering full lifecycle, edge cases, event emission, active count tracking.
- Integration with tick tested in `test/tick.service.test.ts`.

## Observability notes
- Structured log on every action state transition (start, cancel, pause, resume, resolve).
- Action resolution is counted implicitly via tick metrics.
